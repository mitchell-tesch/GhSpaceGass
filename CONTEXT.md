# CONTEXT — GhSpaceGass

Shared vocabulary and domain definitions for the GhSpaceGass Grasshopper plug-in.

---

## Product

**GhSpaceGass** — a Grasshopper 8 plug-in that provides a full round-trip workflow between Grasshopper and SpaceGass via the SpaceGass REST API.

## Domain Terms

### SpaceGass API Service
The local REST API server (`SpaceGassApi.exe`) that provides programmatic access to SpaceGass structural analysis. Runs on `http://localhost:<port>` (default port 34560). Requires a licensed SpaceGass installation.

**Port configuration**: The `--urls http://localhost:<port>` CLI flag is the working method to set a custom port. The `--port` flag documented by SpaceGass does not work in the current beta (v14.50.134-beta1). The plug-in uses `--urls` when launching the service.

### Job
A single SpaceGass project file (`.sg`). The API supports one open job at a time. A job contains structure, loads, analysis settings, and results.

### Singleton Client
A single shared `SpaceGassApiClient` instance that lives for the lifetime of the Grasshopper session. All components access SpaceGass through this instance. No explicit connection wire on the canvas.

### Node
A point in 3D space within a SpaceGass structural model, identified by an integer ID assigned by SpaceGass. Nodes are the endpoints of members and the locations where restraints and point loads are applied.

### Member
A straight structural element connecting two nodes (Node A → Node B). Has a type (beam, truss, cable, etc.), a section, a material, an optional orientation direction, and optional end releases. Polyline geometry produces multiple members sharing intermediate nodes.

### Member Release
A per-end release condition applied to a member, controlling which of the 6 degrees of freedom (Fx, Fy, Fz, Mx, My, Mz) transfer forces through the connection at that end. Each DOF can be Fixed (`F` — force transfers), Released (`R` — no force transfer), or Spring (`S` — partial release with a stiffness value). Represented as a 6-character code string with optional per-DOF stiffness. The `Create Release` builder component uses fixity booleans (true = Fixed, false = Released — same convention as Create Restraint); when a spring stiffness is provided for a DOF, the boolean is ignored and the code is set to `S`. An all-fixed release (`FFFFFF` with no stiffness) is a no-op and is skipped during assembly to optimise API calls. Created by the `Create Release` builder component and assigned to members via the `Create Member` component's optional Release A / Release B inputs (ADR-0013).

### Member Offset
A rigid offset applied at each end of a member, shifting the structural member axis away from the node location. Has X/Y/Z offset values at both end A and end B, in local or global axes (defaults to local). Used to model eccentric connections (e.g., beam connected to column face rather than centreline). Created by the `Create Member Offset` builder component and assigned to members via the `Create Member` component's optional Offset input. Pushed via `Job.Structure.MemberOffsets.Bulk.PostAsync` after members are created. All-zero offsets are skipped to optimise API calls.

### Section
A cross-section profile assigned to members. Can be sourced from the SpaceGass library (by name) or user-defined (with explicit geometric properties: area, Ix, Iy, etc.).

### Material
A material definition assigned to members. Can be sourced from the SpaceGass library (by name) or user-defined (with explicit properties: E, G, density, fy, etc.).

### Restraint
A boundary condition applied at a node, defining which of the 6 degrees of freedom (Fx, Fy, Fz, Mx, My, Mz) are fixed, free, or spring-supported.

### Node Constraint
A master-slave rigid link between two nodes, constraining specified degrees of freedom of the slave node to follow the master node. Has a 6-character constraint code (F=Constrained, R=Free) for the 6 DOFs (TX, TY, TZ, RX, RY, RZ). Default `FFFFFF` = fully rigid link. Supports global or inclined axes (with optional direction vector for inclined). Created by the `Create Node Constraint` builder component and pushed via `Job.Structure.NodeConstraints.Bulk.PostAsync`. Assembled after restraints, before loads.

### Plate
A 3 or 4 node shell/plate finite element. Triangular plates have 3 corner nodes; quadrilateral plates have 4. Has a material (same Material Goo as members), actual thickness, optional bending/membrane/shear thickness overrides, an offset from the nodal plane, and a plate theory (Kirchoff thin plate or Mindlin thick plate). Created by the `Create Plate` builder component from Mesh geometry — each mesh face becomes one plate element. Pushed via `Job.Structure.Plates.Bulk.PostAsync`. Plate corner nodes are deduplicated with member endpoints in the shared node pool. Materials are deduplicated across both members and plates.

### Load Case
A named container for a set of loads (e.g., "Dead Load", "Live Load"). Identified by an integer ID assigned by SpaceGass. The API supports multiple types: Primary and Combination. Primary load cases hold direct loads; Combination load cases reference other load cases with scale factors.

### Combination Load Case
A load case that combines other load cases (primary or combination) with scale factors (e.g., 1.2×Dead + 1.5×Live, or 1.0×ULS + 0.7×Wind_Combo). Created via the `Create Combination Load Case` builder component, which accepts both primary Load Cases and other Combination Load Cases as constituents. The API accepts a title, an optional note, and a list of constituent (load case ID, factor) pairs. Only linear combination is supported. Assembled after primary load cases in dependency order; when combinations reference other combinations, they are created in topological order (dependencies first).

### Load Category
A classification label applied to loads to indicate their source or purpose (e.g., "Dead", "Live", "Wind"). Used for filtering and code-compliance grouping. Optional — node loads may reference a category but are not required to. Created via the API and referenced by ID on individual load entries.

### Node Load
A concentrated force and/or moment applied at a specific node, within a specific load case. Defined in global axes.

### Member Distributed Load
A distributed force and/or moment applied along the length of a member, within a specific load case. Defaults to local member axes; optionally applied in global axes. Forces (Fx, Fy, Fz) and moments (Mx, My, Mz) can be specified at start and end of the loaded region independently — set different values for trapezoidal loading. Forces and moments are separate API types: forces pushed via `Job.Loads.MemberDistributedLoads.Bulk.PostAsync`, moments pushed via `Job.Loads.MemberDistributedMoments.Bulk.PostAsync`. A single `Create Member Distributed Load` component handles both; the assembler splits them into the correct API calls.

### Member Concentrated Load
A concentrated force and/or moment applied at a specific position along a member, within a specific load case. Has force components (Fx, Fy, Fz), moment components (Mx, My, Mz), a position along the member, position units (Percent or Actual), and axis system (Local or Global — defaults to Local, ADR-0011). Created by the `Create Member Concentrated Load` builder component and pushed via `Job.Loads.MemberConcentratedLoads.Bulk.PostAsync`.

### Member Prestress Load
An axial prestress force applied to a member, within a specific load case. Used to model post-tensioning or pre-compression in structural members. Has a single `Prestress` value representing the axial force. Created by the `Create Member Prestress Load` builder component and pushed via `Job.Loads.MemberPrestressLoads.Bulk.PostAsync`.

### Plate Pressure Load
A uniform pressure applied to a plate element, within a specific load case. Has pressure components (Px, Py, Pz) in local or global axes (defaults to local, ADR-0011). References a plate element via the Plate Goo input — corner nodes are used to resolve the plate ID during assembly. Created by the `Create Plate Pressure Load` builder component and pushed via `Job.Loads.PlatePressureLoads.Bulk.PostAsync`.

### Thermal Load
A temperature change and/or thermal gradient applied to a member or plate element, within a specific load case. Has a uniform temperature change, Y thermal gradient, and Z thermal gradient. Applies to both members and plates — the same API type (`ThermalLoadCreate`) with an `ElementType` discriminator (Member or Plate). Created by the `Create Thermal Load` builder component, which auto-detects the element type from the input (Line → member, Plate Goo → plate). Pushed via `Job.Loads.ThermalLoads.Bulk.PostAsync`.

### Self-Weight Load
An automatic gravity load applied to all members based on their section area, material density, and a gravity factor, within a specific load case.

### Lumped Mass Load
A concentrated mass applied at a specific node, within a specific load case. Used for dynamic frequency analysis to represent non-structural mass (e.g., equipment, floor mass) that doesn't come from member self-weight. Has translational mass components (Tmx, Tmy, Tmz) and rotational mass components (Rmx, Rmy, Rmz). Created by the `Create Lumped Mass Load` builder component and pushed via `Job.Loads.LumpedMassLoads.Bulk.PostAsync`.

### Prescribed Displacement
An imposed displacement and/or rotation applied at a specific node, within a specific load case. Used to model support settlements or forced displacements (e.g., differential settlement of a foundation). Has translation components (Tx, Ty, Tz) and rotation components (Rx, Ry, Rz). Created by the `Create Prescribed Displacement` builder component and pushed via `Job.Loads.NodeDisplacements.Bulk.PostAsync`.

### Assemble Model
The compile step that collects all in-memory Goo objects (nodes, members, sections, materials, restraints, loads), deduplicates coincident geometry, resolves geometry-to-ID mappings, and pushes the model to SpaceGass via bulk API calls in dependency order. Supports two modes: Rebuild (default — clears existing data first) and Append (adds to existing model content without clearing).

### Disassemble Model
The reverse of Assemble Model: reads an existing SpaceGass model from the open job via List GET endpoints and outputs the structural geometry (Points, Lines, Meshes) plus a populated SgModel for downstream chaining to Run Analysis and Results. Does not modify the job — read-only.

### SgModel (Model Object)
The Goo type output by both Assemble Model and Disassemble Model. Encapsulates the ID ↔ geometry mappings and job context. Flows downstream to Analysis and Results components.

### Builder Component
A synchronous Grasshopper component that constructs an in-memory Goo object (e.g., SgNode, SgMember) without making any API calls. Pure data construction.

### Async Component
A Grasshopper component that performs API calls on a background thread using the GrasshopperAsyncComponent pattern. The canvas remains responsive during execution. Used by all components that make API calls — Connect, Assemble Model, Disassemble Model, Save Job, Job Info, Run Analysis, all Results components, and all Get query components. Runtime messages are deferred to the SetData phase (UI thread) via `WorkerInstance.AddRuntimeMessage` to survive the Grasshopper re-solve message clear.

### Clear & Rebuild
The default strategy used by Assemble Model: clear all existing model data in the open job (via `Job.Data.Delete(force=true)`) and push the entire model from scratch. The Grasshopper graph is the single source of truth (see ADR-0001). An optional Append mode (Mode input = 1) skips the clear, adding data alongside existing job content — useful for augmenting a model opened via Connect.

### Orphan Point
A Point3d referenced by a restraint or load that does not coincide (within tolerance) with any member endpoint. Assemble Model creates a standalone node for it and emits a warning (see ADR-0002).

### Coincidence Tolerance
The distance threshold used to determine whether two points are the same node. Defaults to the Rhino document tolerance. Configurable.

### Deduplication
The process by which Assemble Model identifies coincident points (within tolerance) and merges them into a single node, ensuring shared connectivity between members, restraints, and loads.

### Analysis Settings
A configuration object that controls analysis behaviour for non-linear static and buckling analyses. Created by the `Create Analysis Settings` builder component and fed as an optional input to `Run Analysis`. When omitted, SpaceGass defaults apply. Settings include number of buckling modes and other solver parameters. Applied to the job immediately before each analysis run (ADR-0014).

### Intermediate Member Forces
Forces (Fx, Fy, Fz, Mx, My, Mz) at stations along the length of a member, not just at the end nodes. Queried via the API's `List Member Intermediate Forces` endpoint. Output as a two-level data tree `{load_case; member}` with station results ordered by position (ADR-0015).

### Intermediate Member Displacements
Translations (TxGlobal, TyGlobal, TzGlobal, TxLocal, TyLocal, TzLocal) at stations along the length of a member. Queried via the API's `List Member Intermediate Displacements` endpoint, which provides global and local translations — no rotations. Output as a two-level data tree `{load_case; member}` with station results ordered by position (ADR-0015).

### Buckling Load Factor
A scalar multiplier indicating the load level at which elastic buckling occurs for a given load case and mode. Returned per load case and mode from the API's `List Buckling Load Factors` endpoint. Output in a data tree `{load_case; mode}`.

### Effective Length
The equivalent length used for buckling capacity calculations, returned per member, per load case, per mode, per axis (Ley, Lez). Also includes the critical buckling load (Pcr) and member length. Queried via the API's `List Buckling Effective Lengths` endpoint. Output in a data tree `{load_case; mode}` with member results as list items within each branch.

### Natural Frequency
A vibration frequency (in Hz) at which the undamped structure oscillates freely, one value per mode returned by a dynamic frequency analysis. Each mode also carries a natural period (seconds) and per-axis mass-participation ratios (Mass Part. X/Y/Z) that indicate how much of the total mass moves in each global direction for that mode. Queried via the API's `List Natural Frequencies` endpoint. Output in a data tree `{load_case; mode}`.

### Mode Shape
The relative deformed shape of the structure at a specific natural frequency. Each mode shape is a list of per-node displacements — translations (Tx, Ty, Tz) and rotations (Rx, Ry, Rz) in the global axes — that describe how the structure vibrates in that mode. Values are eigenvector-relative (no physical units), so only the shape and the ratios between nodes are meaningful. Queried via the API's `List Mode Shapes` endpoint. Output in a data tree `{load_case; mode}` with per-node values as list items. Visualised by `Get Dynamic Frequency Results` as one deformed member line per (load case, mode, member), scaled to a fraction of the model bounding box (ADR-0009).

### Plate Element Forces
Average in-plane forces (Fx, Fy, Fxy), bending moments (Mx, My, Mxy, MxTop, MxBtm, MyTop, MyBtm), and transverse shears (Vxz, Vyz) for a plate element at a specific load case. One result per plate per load case. Queried via `List Plate Element Forces` endpoint. Output in a data tree `{load_case}` with plate results as list items.

### Plate Nodal Forces
Per-node forces (Fx, Fy, Fz) and moments (Mx, My, Mz) at each corner node of a plate element for a specific load case. Multiple results per plate (one per corner node). Queried via `List Plate Nodal Forces` endpoint. Output in a data tree `{load_case; plate}` with node results as list items within each branch.

### Steel Member Check Summary
The SpaceGass steel-design engine's aggregated check for each designed **steel design group** — one record per group reporting the *critical* load case (the one that governs the design), the design capacity ratio (Load Factor = **capacity / action**, so values ≥ 1 indicate an adequate group and values < 1 indicate an overloaded / failed group), a status flag ("PASS"/"FAIL"), the failure mode description, the section name, yield stress, critical segment length, and total member length. Steel design must be run in SpaceGass before this query returns data. Queried via `Job.Query.Design.SteelMember.CheckSummary.GetAsync`. The API's `Member` field on `SteelCheckSummary` is a *design-group* identifier (aggregating one or more physical members that share a steel-design definition), not a physical member ID — this domain model exposes it as `DesignGroupId` and the component labels its outputs accordingly (per-member resolution is a deferred slice). Because the design engine has already collapsed the load-case dimension internally, results are output as **flat parallel lists ordered by design-group ID** rather than as a data tree by load case (ADR-0016). Visualised as coloured representative-lines using a traffic-light ramp on capacity ratio (Deep Red severely overloaded at 0.5 → Red at capacity 1.0 → Orange 1.11 → Yellow 1.33 → Green ≥ 2.0 safe margin).

### Moving Load
A single "row" inside a moving load scenario's Loads list — one vehicle **or** one pressure travelling along one travel path, with optional overrides: `Speed` (metres per second — set to 0 for a stationary load), `Start Position` (metres from the first station along the travel path, must be zero or positive), `Delay` (seconds — only meaningful when the parent scenario has a Time Interval set), `Load Factor`, `Lane Factor`, `Dynamic Factor`, and `Generate Stationary Load Case` mode. The `Generate Stationary Load Case` mode only applies when Speed is 0: `Starting Load Case Only` puts the stationary load into the scenario's Starting Load Case and shifts the moving-load cases to begin in the next case, while `All Load Cases` combines the stationary load into every generated case starting from the Starting Load Case. Wired into a scenario via the `Create Moving Load` builder component and the scenario's `Loads` input. Assemble Model discovers vehicles, pressures and travel paths by walking each scenario's Loads[] — scenarios are the single entry point for the moving-load workflow. A moving load must reference exactly one of Vehicle or Pressure; connecting both, or neither, is rejected at build time.

**Reference-equal resource sharing (ADR-0006 nuance).** When the same Vehicle / Pressure / Travel Path Goo instance is wired into multiple moving loads — the natural way to model one truck running on several paths, or several trucks running on the same path — the assembler collapses it silently to a single API call. This is a deliberate refinement of ADR-0006: reference-equal duplicates are the same declaration reused, so no warning is emitted. Two *different* instances that happen to share a Name still trigger the standard ADR-0006 dedup warning.

**Analysis effect.** Wiring a moving load into a scenario and running Assemble Model stages the moving-load setup in the SpaceGass job. A separate Generate Moving Loads step (coming in a future release) is required before analysis actually produces the generated moving-load load cases — the scenario alone does not create analysable loads.

### Moving Load Elements To Load
The job-level selection of which members and plates receive the distributed wheel or pressure loads during a Generate run. Set per subgroup by `Generate Moving Loads` immediately before each Generate call: the component collapses the branch's `Members To Load` and `Plates To Load` DataTrees into SpaceGass selection strings (adjacent runs written as `N-M` ranges) and pushes them via `Job.Loads.MovingLoads.ElementsToLoad.PatchAsync`. Branches with neither members nor plates are skipped with a warning; branches with no scenarios to apply are also skipped with a warning (SpaceGass would otherwise generate nothing for that subgroup). **Not restored after the run** — the job is left with whatever selection the last processed branch wrote. This matches Moving Load Settings' PATCH-persistence behaviour (see the Settings term for the same interaction with Rebuild mode). A future release will offer a "reset Elements To Load" workflow alongside the reset-to-defaults query for Settings.

### Moving Load Settings
Job-level configuration for the SpaceGass moving-load engine. All fields are individually optional — a field the user leaves disconnected keeps its current SpaceGass value (only the fields you connect are updated). Fields (with SpaceGass meanings verified with the domain owner):
- **Apply To Closest Member** — when true, each wheel load is applied only to the single closest surrounding member (instead of being distributed across the nearest members proportionally to distance). Fewer generated loads, still accurate enough for most cases.
- **Check Vertical Proximity** + **Vertical Proximity** (metres) — when true, wheel loads are only distributed to members / plates whose vertical distance from the load is within the given proximity. Prevents loads on an upper bridge deck being incorrectly applied to a lower deck below it.
- **Ignore Loads On One Member** — when true, wheels or pressure parts that would transfer their load to only one member are ignored. Useful for deactivating wheels that have moved off the side of a deck or off the end of a skew bridge, even while the vehicle is still within the travel-path extents.
- **Ignore Outside Loaded Area** — when true, wheels or pressure parts that fall outside the SpaceGass loading-area polygon are treated as inactive. The loading-area polygon itself is defined graphically inside SpaceGass — this plug-in does not push a loading area in the current release. If no loading-area polygon has been defined in SpaceGass, this flag has no effect.
- **Keep Loads Within Travel Path** — when true, each load starts with its rear at the start of the travel path and finishes when its front reaches the end (crane-rail behaviour). When false, loads approach the start of the path, move along it, and disappear off the end (road-traffic / conveyor behaviour).
- **Retain Loads** — when true, previously-generated loads for scenarios that are deselected in the SpaceGass scenario tree are kept. When false, loads for deselected scenarios are cleared on the next Generate run.

Created by the `Create Moving Load Settings` builder component. Since Slice 43.4 the PATCH is pushed by **`Generate Moving Loads`** rather than `Assemble Model` — settings are a concern of the generation run, not the model definition. Wire the builder's output into Generate's `Settings` input; when omitted, SpaceGass keeps its current settings. When no fields are set on the builder (`HasAnyValue == false`), Generate silently skips the PATCH.

**Interaction with Rebuild mode (ADR-0001).** Moving-load settings are pushed via PATCH: fields the user does not connect are left at their existing SpaceGass value. That value survives `ClearJobDataAsync`, so a setting changed by one Generate run persists in the SpaceGass job until it is *explicitly* overridden — even if the user later disconnects the settings input entirely. This is a deliberate consequence of PATCH semantics; the alternative (send every field on every run, using best-guess defaults for anything the user hasn't set) would require the plug-in to duplicate the SpaceGass default table and would drift out of date. If a user wants to reset moving-load settings back to SpaceGass defaults, they need to do it inside SpaceGass or explicitly set every field on this component.

**Follow-ups.** (a) The API's `LoadedArea` polygon field (a spatial JSON blob) is not exposed by this plug-in yet — a future release will introduce a builder for it once the shape is well-typed. (b) A `Get Moving Load Settings` query component that reads back the job's current settings, plus a similar "reset to defaults" workflow for both Settings **and** the Moving Load Elements To Load selection, are on the same future-release list as the other read-back gaps (existing combination-case IDs, library discovery for vehicles / sections / materials).

### Moving Load Generation
The terminal step of the moving-load workflow — the SpaceGass moving-load engine reads each scenario's Loads (Vehicle/Pressure + Travel Path + factors) and produces a set of **generated moving-load load cases**, one per station along each travel path. Runs via the `Generate Moving Loads` async component (`SpaceGass > 5 | Loads`). Inputs are a Model (from Assemble Model), three matching DataTrees describing the run — `Members To Load`, `Plates To Load`, and `Scenarios To Apply` — plus optional `Moving Load Settings` and `Load Category`. Each branch path in the trees is one **subgroup**: for that subgroup, the component sets the Moving Load Elements To Load selection (which members / plates receive load), toggles the branch's scenarios to `Include = true` (and disables all other scenarios that appear anywhere in the input tree), then runs Generate. **Tree matching:** the component iterates the *union* of branch paths across the three trees — a branch that only exists in one tree is still attempted, with the missing trees treated as empty for that branch (so a branch with no members and no plates, or no scenarios, is skipped with a warning rather than silently doing nothing). The generated load-case IDs and group labels SpaceGass returns are captured per branch on the output DataTrees. Every scenario Include flag that the run touches is snapshotted before the run and restored afterwards — even on exception — so the job state after Generate matches the state before, apart from the generated load cases themselves. **However**, the Elements To Load selection is *not* restored — the job is left with whatever selection the last processed branch wrote (see the Moving Load Elements To Load term). Load Category (when supplied) tags every generated case with that category. Selection strings are formatted using `ModelAssembler.FormatIdSelectionString`, which collapses adjacent runs into `N-M` ranges.

### Moving Load Scenario
The container that references vehicles, pressures and travel paths (via the `Loads` input introduced in Slice 43.1) and defines a moving-load run along a travel path. When the SpaceGass moving-load engine acts on a scenario it produces a **generated moving-load load case** for each station along the path — a normal SpaceGass load case that shows the structure's response with the vehicle/pressure at that station. Note that the generated load cases only appear in the job once a separate `Generate Moving Loads` step is run against the assembled scenarios — assembling a scenario alone stages the setup but does not produce load cases (Generate lands in a future release). Each scenario has a Name, an optional `Loads` list (one or more Moving Loads — each a vehicle or pressure travelling along a travel path), an optional Starting Load Case (anchoring the first generated load case), optional Time Interval (the time step between generated load cases when the analysis engine treats the scenario as a time series, > 0), Include flag (default true), and an optional list of Combination entries. Each Combination entry references a load case (primary or combination) plus two factors — LoadCaseFactor (scale for the combined LC) and ScenarioFactor (scale for the moving-load contribution) — and an optional StartingCombinationCase integer that pins the starting ID of the resulting combination cases (used when appending to an existing model to avoid ID collisions; null → server-assigned; deliberate exception to the Server-assigned IDs principle per ADR-0017). Created by the `Create Moving Load Scenario` builder component and pushed via `Job.Loads.MovingLoads.Scenarios.Bulk.PostAsync`, then the Loads[] is PUT via `Job.Loads.MovingLoads.Scenarios.Item[<id>].Loads.PutAsync`. Deduplicated by Name (ADR-0006). Assembled after primary + combination load cases (scenarios reference their IDs) and before load categories. Scenarios with no Loads[] entries still get created but Assemble Model warns that they will produce no generated load cases.

### Moving Load Vehicle
A reusable vehicle template that the SpaceGass moving-load engine drags along a travel path. Two flavours: **library** (only Library key + Name; SpaceGass supplies the wheel layout and units from a named vehicle library) and **user-defined** (a list of Wheel Loads — position `X, Y` and force / moment components `Fx, Fy, Fz, Mx, My, Mz` — plus Force / Length / Moment units for the load values). Wheel `X` is the position along the vehicle (values are conventionally negative — wheels trail behind the vehicle's leading-edge reference point); Wheel `Y` is the transverse position across the vehicle width (maps onto the model's global Z axis in Y-vertical models and onto global Y in Z-vertical models). Wheel forces and moments are always applied in the model's **global** axes — which axis is vertical depends on the SpaceGass job's Vertical Axis setting (visible on `Job Info`). Created by the `Create Moving Load Vehicle` builder component; user-defined vehicles are pushed via `Job.Loads.MovingLoads.Vehicles.Bulk.PostAsync`, library-sourced vehicles via `Job.Loads.MovingLoads.Vehicles.Library.PostAsync` one at a time (the SpaceGass library endpoint is not bulk). Deduplicated by `Library::Name` (library-sourced) or `Name` (user-defined) — same convention as `Create Section` / `Create Material` (ADR-0006). Assembled before moving load scenarios so that scenario `Loads[]` entries — populated in a future release — can resolve vehicle IDs. **UX note:** wheel loads are entered as parallel `Wheel X` / `Wheel Y` / `Wheel Fx…Mz` lists on the builder rather than through a dedicated wheel-load sub-component; the trade-off is a compact but wide builder that requires the user to keep all populated lists the same length (the builder validates and reports a clear error on mismatch). Revisit the sub-component option if user feedback shows the parallel-list shape is hard to maintain for large vehicles.

### Moving Load Pressure
A reusable rectangular pressure patch (Width × Length) that the SpaceGass moving-load engine drags along a travel path. `Width` is the transverse dimension (aligned with Wheel Y on vehicles); `Length` is the along-vehicle dimension (aligned with Wheel X). Has per-axis pressure components (Px, Py, Pz) in the model's **global** axes (same vertical-axis convention as vehicles) and an optional Load Spacing — the grid resolution SpaceGass uses to discretise the pressure patch into a set of point loads (smaller values produce a finer approximation of the uniform pressure). Pressures are always user-defined; there is no SpaceGass library equivalent. Created by the `Create Moving Load Pressure` builder component and pushed via `Job.Loads.MovingLoads.Pressures.Bulk.PostAsync`. Deduplicated by Name (ADR-0006). Assembled alongside vehicles, before scenarios.

### Vehicle Wheel Load
A single load applied at a point on a vehicle's footprint — position (X, Y) on the vehicle template plus optional force (Fx, Fy, Fz) and moment (Mx, My, Mz) components. In this plug-in the wheel-load data is a purely internal record inside `SgMovingLoadVehicleData`; users define wheel loads via the `Wheel X` / `Wheel Y` / `Wheel Fx` … `Wheel Mz` parallel lists on `Create Moving Load Vehicle` rather than through a dedicated component. The convention: `X` is along the vehicle (conventionally negative — wheels trail behind the reference point), `Y` is transverse across the vehicle width, and `Fx / Fy / Fz` / `Mx / My / Mz` are always in the model's global axes (vertical axis depends on the job's Vertical Axis setting).

### Moving Load Travel Path
A named polyline the SpaceGass moving-load engine drags a vehicle or pressure along. Made up of two or more **Stations** in polyline vertex order — **the polyline's direction is the travel direction**, so the vehicle moves from vertex 0 to vertex N (use Grasshopper's `Flip Curve` upstream if the polyline runs the wrong way). Created by the `Create Moving Load Travel Path` builder component from any `Line` or `Polyline` (accepted-geometry convention per ADR-0003 — arcs and free-form curves must be discretised first). Stations are pushed at the absolute coordinates of each vertex — the plug-in does not snap them to nearby model nodes. Pushed as a two-step API call: the path definition (name only) via `Job.Loads.MovingLoads.TravelPaths.Bulk.PostAsync`, then the station list per path via `Job.Loads.MovingLoads.TravelPaths.Item[<id>].Stations.PutAsync` (the SpaceGass stations endpoint is not bulk). Deduplicated by Name (ADR-0006). Assembled after moving load vehicles and pressures, before scenarios, so that scenario `Loads[]` entries — populated in a future release — can resolve travel-path IDs.

### Moving Load Station
A single point along a moving load travel path. The station's `Position` is an absolute point in the model's coordinate space — it is *not* a reference to any structural node (the API's `NodeKey` field is the station's own identifier local to the travel path and is assigned by SpaceGass, so the plug-in always leaves it null). Every station carries an optional `Radius` — in **metres** — that describes the **arc replacing the straight chord from the previous station to this one**: the chord endpoints stay pinned and the arc of that radius bulges to one side of the chord (a negative radius bulges to the opposite side). The first station in a path has no previous chord and its radius is ignored by SpaceGass (the builder still accepts a value there without erroring — supply any placeholder). When Radius is null the segment is a straight chord.

## Architecture References

Detailed decisions are recorded in `adr/0001–0017`. Key choices:

- **Round-trip workflow**: GH → SpaceGass (push model) → run analysis → GH (pull results)
- **Grasshopper tab**: `SpaceGass` (no space)
- **Panel layout**: `Connection` (Connect, Job Info, Save Job), `Properties` (Create Section, Create Material, Get Sections, Get Materials), `Structure` (Create Member, Create Restraint, Create Release, Create Member Offset, Create Node Constraint, Create Plate), `Cases` (Create Load Case, Create Combination Load Case, Create Load Category, Get Load Cases), `Loads` (Create Node Load, Create Member Distributed Load, Create Member Concentrated Load, Create Member Prestress Load, Create Self-Weight Load, Create Lumped Mass Load, Create Prescribed Displacement, Create Plate Pressure Load, Create Thermal Load, Create Moving Load Scenario, Create Moving Load, Create Moving Load Vehicle, Create Moving Load Pressure, Create Moving Load Travel Path, Create Moving Load Settings, Generate Moving Loads, Get Self-Weight Loads, Get Node Loads, Get Member Loads, Get Plate Loads), `Model` (Assemble Model, Disassemble Model), `Analysis` (Create Analysis Settings, Run Analysis), `Results` (Get Node Reactions, Get Node Displacements, Get Member Forces, Get Member Displacements, Get Buckling Results, Get Dynamic Frequency Results, Get Plate Forces, Get Steel Member Check Summary)
- **Library vs Custom naming**: Unified components — `Create Section` and `Create Material` handle both library (when Library input is connected) and custom (when Library is omitted) modes. Same Goo types for both.
- **Fine-grained components**: one component per concept
- **Deduplication (ADR-0006 + nuance)**: Assemble Model deduplicates entities by key (Library::Name for sections/materials/library vehicles, Name for everything else) and emits a warning listing the dropped duplicates. Slice 43.1 adds one nuance for the moving-load resources discovered through scenarios: **reference-equal** duplicates (the same Vehicle / Pressure / Travel Path Goo instance wired into multiple moving loads) collapse silently to a single API call with no warning, because that is the natural way to model one truck running on several paths or several trucks running on the same path. Different instances with the same Name still warn as ADR-0006 specifies.
- **Deferred push**: builder components produce Goo; Assemble Model compiles and sends
- **Server-assigned IDs**: no manual ID allocation; SpaceGass returns IDs on creation. Documented exception: user-facing integer inputs are permitted when the field refers to pre-existing content the plug-in did not create (Append-mode workflows) AND no Goo abstraction can stand in AND the default value preserves the server-assigned behaviour. `Starting Combination Cases` on `Create Moving Load Scenario` is the current instance (ADR-0017).
- **Async via GrasshopperAsyncComponent**: source-included, not NuGet
- **Results**: analysis results output data trees with Load Cases, Node IDs, and Member IDs in matching tree structures for easy correlation. Static results use `{load_case}` for node results, `{load_case; member}` for member results, `{load_case}` for plate element forces, and `{load_case; plate}` for plate nodal forces (ADR-0008, ADR-0015). Buckling and dynamic frequency results use `{load_case; mode}`. **Design results (steel design, etc.) output flat parallel lists ordered by element ID rather than a load-case tree** — the critical load case is a value on each record, not a dimension (ADR-0016). All results components include Load Cases filter input supporting both primary and combination load case names (design components filter only by element ID since load case is aggregated). Node/Member/Plate filter inputs accept integer IDs (from Assemble Model or Disassemble Model outputs). Geometry outputs (Points, Lines, Meshes) on first branch only to avoid heavy GH preview of repeated data. Viewport preview with auto-scale (ADR-0009): Visual Scale ("VSc") input controls preview size (auto-scale when omitted, 0=off); Show Values? ("SV?" on all preview-capable components except Get Plate Forces, which uses "SV" as a pre-existing outlier) toggles numeric labels. Forces and moments auto-scale independently (different units). Auto-scale uses only filtered results geometry (not full model) for correct proportions when filtering. Visual ("V") selector on Get Member Forces (Fx–Mz, default Mz, None=off) and Get Plate Forces (Fx–MyBtm, default Mx, None=off) selects which component to display. Preview types: reaction arrows (per-axis RGB), displacement vectors (purple), displaced shape polylines (cyan), force diagrams (per-component colours with extrema labels), plate contours (Turbo spectrum colormap), mode shapes (per-mode palette with optional "Mode N (X Hz)" labels), steel capacity-ratio coloured representative-lines (traffic-light ramp: Deep Red severely overloaded at 0.5 → Red at capacity 1.0 → Orange at 1.11 → Yellow at 1.33 → Green safe at ≥ 2.0; LoadFactor = capacity/action, so &lt; 1 is overloaded). Preview geometry pre-computed during SetData — zero per-frame allocation. Shared helpers: `PreviewDrawHelper` (arrowheads, moment arcs), `SgPoint3DExtensions.ToPoint3d()`, `SgModelData.BuildLoadCaseIdToNameMap()`.
- **Error handling**: all async components use `ModelAssembler.FormatApiError` to extract meaningful messages from `ErrorResponse` (Title, Detail, HTTP status code) and `ApiException`. Errors are surfaced as both a component-level error badge (via `AddRuntimeMessage`) and in the Status output.
- **Member releases**: separate builder component (ADR-0013)
- **Analysis settings**: builder Goo fed into Run Analysis (ADR-0014)
- **Design results output structure**: flat parallel lists per element (ADR-0016)
- **Manual ID allocation exception for Append workflows**: user-facing integer ID inputs permitted only for pre-existing content the plug-in did not create (ADR-0017)
- **User-facing wording**: component names, nicknames, tooltips, error messages and warnings use the domain vocabulary defined in this document. They MUST NOT reference internal delivery structure (slice numbers, upcoming slices, backlog phrasing) or implementation-language jargon ("API type", "domain model", "Goo", "canonical", etc.). Forward-looking mentions in user-facing strings use user-facing language like "a future release" or "not yet supported".
